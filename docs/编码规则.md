# 编码规则

行为准则，减少 LLM 编码中的常见错误。结合项目具体需求一并使用。

**权衡：** 这些准则偏向谨慎而非速度。简单任务请自行判断。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。主动呈现权衡。**

编码前：

- 明确说出你的假设。不确定的地方，直接问。
- 存在多种解释时，呈现出来——不要默默选一个。
- 存在更简单的方案时，指出它。该反驳就反驳。
- 有不清楚的地方，停下来。说出哪里困惑，再问。

## 2. 简洁优先

**用最少的代码解决问题。不要写 speculative 的代码。**

- 不添加需求之外的功能。
- 不为单次使用的代码创建抽象。
- 不添加没有要求的"灵活性"或"可配置性"。
- 不处理不可能发生的错误场景。
- 如果你能写 200 行但其实 50 行就够，重写。

问自己："一个资深工程师会觉得这过度设计吗？"如果会，简化。

## 3. 精准修改

**只改必须改的。只清理自己造成的烂摊子。**

修改现有代码时：

- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的地方。
- 保持现有风格，即使你个人风格不同。
- 发现无关的死代码，提一句——不要擅自删除。

当你的修改产生了孤立代码时：

- 移除因本次修改而不再使用的 import/变量/函数。
- 不要删除原本就存在的死代码，除非被要求。

检验标准：每一行修改都要能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义可验证的成功标准。循环验证直到达成。**

把任务转化为可验证的目标：

- "添加验证" → "先写无效输入的测试，再让测试通过"
- "修复 bug" → "先写能复现 bug 的测试，再让测试通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务，先简述计划：

```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

明确的成功标准让你可以独立循环验证。模糊的标准（"让它能跑"）需要不断确认。

---

**准则生效的表现:**
diff中减少不必要变更、减少因过度设计导致的返工、澄清性问题在实现前而非实现后出现。
